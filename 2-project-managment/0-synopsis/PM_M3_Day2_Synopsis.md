# ⏰ **9:10-9:30 (20 min) - WORK BREAKDOWN STRUCTURE (WBS) - TEORIJA**

### **Slajdovi 17-20: What is WBS? + Why is WBS Important?**

**Notes iz PDF-a:**
_"A Work Breakdown Structure (WBS) is a hierarchical decomposition of a project into smaller, manageable components. It visually organizes the project scope into phases, deliverables, and tasks, ensuring that all work is clearly defined."_

## **Što reći studenticama (20 min):**

"Ulazimo u jedan od NAJVAŽNIJIH alata u project managementu - **Work Breakdown Structure** ili skraćeno **WBS**.

---

🧩 **ŠTO JE WBS?**

**Definicija:**
> Work Breakdown Structure (WBS) je hijerarhijska dekompozicija projekta na manje, upravljive komponente.

**Jednostavnije rečeno:**
WBS je način da razbijete VELIKI projekt na MALE dijelove koje je lakše upravljati.

**Analogija - Puzzle:**
Zamislite da vam netko da puzzle od 1000 dijelova u kutiji i kaže: 'Složi to!'

Kako biste pristupili?
1. Prvo sortirate po bojama/dijelovima
2. Zatim radite rubove
3. Zatim središnje dijelove
4. Na kraju spajate sekcije

WBS radi ISTO s projektom - razbija ga na logične dijelove!

---

📊 **STRUKTURA WBS-a:**

*[Nacrtaj na whiteboard]*

```
                    PROJEKT
                       │
        ┌──────────────┼──────────────┐
        │              │              │
     FAZA 1        FAZA 2         FAZA 3
        │              │              │
    ┌───┼───┐      ┌───┼───┐      ┌───┼───┐
    │   │   │      │   │   │      │   │   │
  Task Task Task  Task Task Task  Task Task Task
```

**3 razine WBS-a:**

| Razina | Naziv | Opis | Primjer (IT) |
|--------|-------|------|--------------|
| **Level 1** | Project | Cijeli projekt | EduTrack Platform |
| **Level 2** | Phases/Deliverables | Glavne faze ili deliverables | Planning, Design, Development, Testing, Launch |
| **Level 3** | Work Packages/Tasks | Konkretni zadaci | Create wireframes, Develop login module |

---

🎯 **ZAŠTO JE WBS VAŽAN?**

*[Prikaži slajdove 18-19]*

**1. Poboljšava jasnoću i organizaciju projekta**
- ✅ Razbija projekt na upravljive dijelove
- ✅ Tim razumije što treba napraviti
- ❌ Bez WBS-a: 'Napravi aplikaciju' → Gdje početi?!

**2. Poboljšava planiranje i scheduling**
- ✅ Svakom zadatku možemo dodijeliti vrijeme i resurse
- ✅ Lakše identificiramo dependencies (koji task ovisi o kojem)
- ❌ Bez WBS-a: Nemoguće napraviti realan raspored

**3. Sprječava Scope Creep**
- ✅ Jasno definirano što JE uključeno
- ✅ Ako nečega nema u WBS-u, nije dio projekta
- ❌ Bez WBS-a: 'Možemo dodati još ovo malo...' × 100

**4. Olakšava raspodjelu resursa**
- ✅ Svaki task ima odgovornu osobu
- ✅ Nema dupliciranja posla
- ❌ Bez WBS-a: 'Mislio sam da ti to radiš!'

**5. Poboljšava procjenu troškova**
- ✅ Svaki work package ima svoj trošak
- ✅ Zbroj = ukupni budget
- ❌ Bez WBS-a: Nasumično pogađanje budgeta

---

📋 **100% RULE - ZLATNO PRAVILO WBS-a**

> **100% Rule:** WBS mora uključiti 100% posla definiranog u Scope-u - ni više, ni manje.

**Što to znači praktično:**
- Ako je nešto u Scope-u → MORA biti u WBS-u
- Ako nečeg NEMA u Scope-u → NE SMIJE biti u WBS-u
- Zbroj svih work packages = cijeli projekt

*[Nacrtaj na whiteboard]*
```
SCOPE STATEMENT says: 'Build login, dashboard, reports'

WBS MUST include:
✅ Login module
✅ Dashboard module  
✅ Reports module

WBS must NOT include:
❌ Mobile app (nije u scope-u)
❌ Payment system (nije u scope-u)
```

📖 **WBS DICTIONARY - Dodatni detalji**

Za svaki work package, WBS Dictionary dodaje:
- **Opis:** Što točno uključuje task
- **Odgovorna osoba:** Tko radi
- **Trajanje:** Koliko vremena treba
- **Troškovi:** Koliko košta
- **Dependencies:** Koji task mora biti gotov prije

*Primjer za 2.1 Create wireframes:*
| Element | Detalji |
|---------|---------|
| ID | 2.1 |
| Naziv | Create wireframes |
| Opis | Kreirati low-fidelity wireframes za sve ekrane aplikacije |
| Odgovorna osoba | UX Designer |
| Trajanje | 5 dana |
| Troškovi | 2.500€ |
| Dependencies | 1.1 Define requirements mora biti gotov |

Ima li pitanja o WBS-u prije nego krenemo na vježbu?"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
WBS je poput "obiteljskog stabla" projekta. Svaki član (task) ima svoje mjesto u hijerarhiji. Parent-child odnos: Design je "roditelj" wireframes-a, wireframes je "dijete" Design-a.

**Ključne stvari za naglasiti:**
1. WBS se čita ODOZGO PREMA DOLJE (od projekta do taskova)
2. Svaki task na Level 3 mora biti dovoljno mali da se može procijeniti (1-10 dana posla)
3. 100% Rule je KRITIČNA - ako nešto nije u WBS-u, NE RADI SE!

**Česta pitanja:**

Q: "Koliko razina treba imati WBS?"
A: "Tipično 3-4 razine. Za male projekte 3 je dovoljno. Za velike kompleksne projekte može biti i 5-6. Pravilo: Toliko razina koliko treba da najniža razina bude upravljiva (1-10 dana posla)."

Q: "Tko kreira WBS?"
A: "PM kreira s timom! PM ne može sam znati sve tehničke detalje. Developer zna bolje koliko treba za backend, Designer za wireframes, itd."

Q: "Što ako nešto zaboravimo staviti u WBS?"
A: "Zato je WBS 'živi dokument' - može se ažurirati. Ali svaka promjena mora proći kroz Change Request Process (učili smo jučer!)."

**Troubleshooting:**
- **Studentice zbunjene hijerarhijom:** Koristi analogiju "obiteljskog stabla" - projekt je "djed", faze su "roditelji", taskovi su "djeca"
- **Pitaju o alatima:** "WBS se može raditi na papiru, Excel-u, MS Project-u, ili specijaliziranim alatima. Za učenje, papir je najbolji!"

_"Project Schedule: A detailed timeline outlining when project tasks and milestones will be completed. Milestone: A significant event or achievement in the project timeline."_

## **Što reći studenticama (20 min):**

"Sada kad imamo WBS, znamo ŠTO treba napraviti. Scheduling nam govori KADA!

📅 **KLJUČNI POJMOVI U SCHEDULINGU:**

---

**1. 📅 PROJECT SCHEDULE**

**Definicija:**
> Detaljan vremenski plan koji pokazuje kada će projektni zadaci i milestones biti završeni.

**Što uključuje:**
- Lista svih taskova (iz WBS-a!)
- Trajanje svakog taska
- Početak i kraj svakog taska
- Međuovisnosti taskova

*IT Primjer:*
'EduTrack projekt ima schedule od 8 mjeseci - od requirements gathering-a u siječnju do lansiranja u kolovozu.'

---

**2. 🏁 MILESTONE (Prekretnica)**

**Definicija:**
> Značajan događaj ili postignuće u vremenskoj liniji projekta.

**Karakteristike:**
- Milestone ima **trajanje = 0 dana** (to je TOČKA, ne period)
- Označava završetak faze ili važan deliverable
- Često vezani uz odobrenja ili review-e

*IT Primjeri:*
- 'Design approved' - dizajn odobren
- 'Beta version released' - beta verzija puštena
- 'Go-live' - lansiranje

*[Nacrtaj na whiteboard]*
```
Timeline: ──────────────────────────────────
          │         │              │
       Start   Milestone 1    Milestone 2
                (Design done)  (Beta ready)
```

---

**3. 📊 BASELINE SCHEDULE**

**Definicija:**
> Originalni odobreni schedule koji se koristi za praćenje napretka.

**Zašto je važan:**
- Baseline je 'originalni plan'
- Uspoređujemo STVARNO stanje s baseline-om
- Ako kasnimo 2 tjedna = 2 tjedna iza baseline-a

*IT Primjer:*
'Baseline kaže da Development završava 15. travnja. Danas je 20. travnja i još nismo gotovi = 5 dana kašnjenja.'

---

**4. 👥 RESOURCE ALLOCATION**

**Definicija:**
> Dodjeljivanje dostupnih članova tima, alata i budgeta specifičnim zadacima.

*IT Primjer:*
'2 developera su assigned na User Authentication modul od 1. do 15. ožujka.'

---

**5. ⏱️ LEAD TIME vs LAG TIME**

Ovo su dva KRITIČNA pojma koja često zbunjuju!

**LEAD TIME (Vrijeme napredovanja):**
> Vrijeme za koje successor task MOŽE POČETI PRIJE nego što predecessor završi.

*Primjer:*
'Editing može početi 2 dana PRIJE nego što je writing potpuno završen.'
```
Writing:  ████████████
Editing:        ████████████
           ←2 dana→ (Lead)
```

**LAG TIME (Vrijeme čekanja):**
> Vrijeme koje MORA PROĆI nakon završetka taska prije nego sljedeći može početi.

*Primjer:*
'Nakon što obojimo zid, moramo čekati 24 sata da se osuši prije drugog sloja.'
```
Painting 1st coat: ████████
                        ┃ 24h lag
Painting 2nd coat:      ████████
```

💡 **Kako zapamtiti:**
- **Lead** = Task može krenuti RANIJE (lead = voditi, biti ispred)
- **Lag** = Task mora čekati (lag = zaostajati)

---

**6. ⏳ FLOAT (Slack Time)**

**Definicija:**
> Vrijeme za koje task može kasniti BEZ da utječe na završetak projekta.

*IT Primjer:*
'Task Documentation ima 3 dana float-a. Može kasniti do 3 dana i projekt će još uvijek završiti na vrijeme.'

**Važno:**
- Taskovi na **Critical Path** imaju **0 float** - ako kasne, kasni cijeli projekt!
- Taskovi s float-om imaju fleksibilnost

---

**7. 📊 GANTT CHART**

*[Prikaži slajd 26]*

**Definicija:**
> Vizualni bar chart koji prikazuje taskove, trajanja i međuovisnosti.

*[Nacrtaj na whiteboard]*
```
Task            | Pon | Uto | Sri | Čet | Pet |
----------------|-----|-----|-----|-----|-----|
1. Requirements | ████|     |     |     |     |
2. Wireframes   |     | ████| ████|     |     |
3. UI Design    |     |     |     | ████| ████|
4. Dev Start    |     |     |     |     |  ◆  |
                                         Milestone
```

**Što Gantt Chart pokazuje:**
- ✅ Koji taskovi se rade kada
- ✅ Koliko traje svaki task
- ✅ Koji taskovi se preklapaju (paralelni rad)
- ✅ Milestones
- ✅ Dependencies (strelice između taskova)

---

📉 **UTJECAJ RESURSA NA SCHEDULE:**

*[Prikaži slajdove 27-28]*

Resursi (ljudi, oprema, alati) DIREKTNO utječu na schedule!

**5 načina kako resursi utječu:**

| Problem | Utjecaj na Schedule | Primjer |
|---------|---------------------|---------|
| **1. Resource Shortage** | Taskovi kasne jer nema tko raditi | Senior developer na bolovanje, backend kasni |
| **2. Overallocation** | Pad produktivnosti, burnout | Developer radi na 3 projekta istovremeno |
| **3. Unused Resources** | Neiskorišten potencijal, viši troškovi | QA čeka 2 tjedna jer development kasni |
| **4. Resource Leveling** | Potrebna redistribucija posla | Preraspodijeliti taskove da svi imaju ravnomjerno |
| **5. Budget Impact** | Treba dodatne resurse = dodatni troškovi | Unajmiti freelancera jer tim ne stiže |

💡 **Ključna poruka:**
Schedule nije samo 'staviti taskove u kalendar'. Mora uzeti u obzir TKO će raditi i KOLIKO je dostupan!

Ima li pitanja o scheduling terminologiji?"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
Lead i Lag su najčešće zbunjujući pojmovi. Koristi primjere iz stvarnog života:
- **Lead:** Možeš početi jesti predjelo dok čekaš da glavno jelo bude gotovo
- **Lag:** Moraš čekati 30 min nakon jela prije nego se kupat

**Critical Path (bonus info ako pitaju):**
Critical Path je najduži put kroz mrežu taskova. Taskovi na tom putu imaju 0 float - svako kašnjenje na Critical Path = kašnjenje cijelog projekta.

# ⏰ **11:15-11:25 (10 min) - BUDGET DEVELOPMENT METHODS - TEORIJA**

### **Slajdovi 30-33: Budget Development Methods**

**Notes iz PDF-a:**
_"Top-Down Budgeting: Senior management sets a total budget based on organizational goals, and the project manager allocates funds to different tasks. Bottom-Up Budgeting: The project team estimates costs for each individual task or activity, then sums them up. Three-Point Budgeting: Uses Optimistic, Most Likely, and Pessimistic estimates."_

## **Što reći studenticama (10 min):**

"Sada imamo WBS (ŠTO) i Schedule (KADA). Ostaje pitanje: **KOLIKO KOŠTA?**

💰 **3 METODE BUDGETIRANJA:**

Postoje 3 glavne metode za procjenu budgeta projekta. Svaka ima svoje prednosti i mane.

---

**1. 📊 TOP-DOWN BUDGETING (Odozgo prema dolje)**

**Pristup:**
> Uprava postavlja UKUPNI budget, a PM ga raspodjeljuje na taskove.

**Kada koristiti:**
- Projekti s FIKSNIM budgetom
- Kada treba BRZA procjena
- Kada nema detaljnih informacija

*IT Primjer:*
'CEO kaže: Imamo 100.000€ za novi CRM. PM raspodjeljuje: 40.000€ development, 20.000€ design, 20.000€ testing, 20.000€ implementacija.'

**Pros i Cons:**
| ✅ Pros | ❌ Cons |
|---------|---------|
| Brzo i jednostavno | Može biti neprecizno |
| Jasna ograničenja | Taskovi mogu biti underfunded |
| Usklađeno s ciljevima firme | Tim nije uključen u procjenu |

---

**2. 📋 BOTTOM-UP BUDGETING (Odozdo prema gore)**

**Pristup:**
> Tim procjenjuje trošak SVAKOG taska, zatim se zbraja u ukupni budget.

**Kada koristiti:**
- Kompleksni projekti koji zahtijevaju PRECIZNOST
- Kada ima vremena za detaljnu analizu
- Kada je tim iskusan i može dati dobre procjene

*IT Primjer:*
```
UI Design: 5.000€
Backend: 20.000€
Frontend: 15.000€
Testing: 10.000€
─────────────────
UKUPNO: 50.000€
```

**Pros i Cons:**
| ✅ Pros | ❌ Cons |
|---------|---------|
| Najtočnija metoda | Dugotrajna |
| Tim je uključen | Može premašiti funding |
| Temelji se na stvarnim podacima | Zahtijeva detaljno poznavanje projekta |

---

**3. 🎯 THREE-POINT BUDGETING (PERT metoda)**

**Pristup:**
> Uzima u obzir 3 scenarija: Optimistični, Najvjerojatniji i Pesimistični.

**Formula:**
```
Expected Cost = (O + 4M + P) / 6

Gdje je:
O = Optimistic (best-case, minimum cost)
M = Most Likely (realistic expected cost)
P = Pessimistic (worst-case, maximum cost)
```

*IT Primjer za Backend Development:*
- **O (Optimistic):** 15.000€ (sve ide savršeno)
- **M (Most Likely):** 20.000€ (normalan tijek)
- **P (Pessimistic):** 35.000€ (problemi, refactoring)

**Kalkulacija:**
```
Expected = (15.000 + 4×20.000 + 35.000) / 6
Expected = (15.000 + 80.000 + 35.000) / 6
Expected = 130.000 / 6
Expected = 21.667€
```

**Kada koristiti:**
- Projekti s VISOKOM NESIGURNOŠĆU
- IT, R&D, inovativni projekti
- Kada postoje rizici koje je teško procijeniti

**Pros i Cons:**
| ✅ Pros | ❌ Cons |
|---------|---------|
| Uzima rizike u obzir | Zahtijeva expert judgment |
| Daje realniju procjenu | Kompleksnija kalkulacija |
| Pomaže u planiranju contingency-ja | Subjektivnost u procjenama O, M, P |

---

💡 **KOJA METODA KADA?**

| Situacija | Preporučena metoda |
|-----------|-------------------|
| Brza procjena, fiksan budget | Top-Down |
| Detaljan, precizan plan | Bottom-Up |
| Nesiguran, rizičan projekt | Three-Point |
| Kombinacija | Često se koristi Bottom-Up + Three-Point |

---

📦 **CONTINGENCY RESERVE (Buffer)**

Uvijek dodajte **10-20% contingency** na procijenjeni budget!

*Zašto?*
- Nepredviđene situacije
- Scope changes (koje odobri sponsor)
- Rizici koji se materijaliziraju

*Primjer:*
```
Procijenjeni budget: 100.000€
Contingency (15%):    15.000€
─────────────────────────────
UKUPNI BUDGET:       115.000€
```

Ima li pitanja o budget metodama prije vježbe?"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
Three-Point formula (PERT) je zapravo weighted average - Most Likely dobiva 4x više težine jer je najvjerojatniji scenarij. Formula dolazi iz PERT (Program Evaluation and Review Technique) metodologije iz 1950-ih (US Navy za Polaris projekt).

**Česta pitanja:**

Q: "Zašto je Most Likely ×4?"
A: "Statistički, Most Likely scenarij se najčešće događa, pa mu dajemo najveću težinu u formuli."

Q: "Koliki contingency treba uzeti?"
A: "Standard je 10-20%. Za nove, rizične projekte: 20-30%. Za poznate, rutinske projekte: 5-10%."

Q: "Što ako Bottom-Up premašuje odobreni budget?"
A: "Onda imate 3 opcije: 1) Smanjiti scope, 2) Tražiti više novca od sponsora, 3) Prihvatiti veći rizik s manjim budgetom."

**Pojednostavljenje formule:**
Ako se studentice muče s matematikom, reci: "U praksi, mnogi alati automatski računaju. Važno je razumjeti da Three-Point uzima u obzir rizike i daje realniju procjenu od jednostavnog prosjeka."

📖 **Reference:**
- [PERT Estimation Explained](https://www.projectmanagement.com/articles/299448/PERT-Estimation)
- [YouTube - Project Budget](https://youtu.be/cJ2WGy013wq)

## ✅ **ASSESSMENT CRITERIA**

### **Razumijevanje WBS-a:**
| Kriterij | Početnik | Srednji | Napredni |
|----------|----------|---------|----------|
| Struktura | WBS ima samo 2 razine | WBS ima 3 razine | WBS ima 3+ razine s jasnom logikom |
| 100% Rule | Neki deliverables nedostaju | Većina pokrivena | Svi deliverables pokriveni |
| Numeracija | Nema konzistentne numeracije | Djelomična numeracija | Potpuna konzistentna numeracija |

### **Razumijevanje Schedulinga:**
| Kriterij | Početnik | Srednji | Napredni |
|----------|----------|---------|----------|
| Terminologija | Poznaje 2-3 pojma | Poznaje većinu pojmova | Može objasniti sve pojmove |
| Gantt Chart | Samo taskovi bez vremena | Taskovi s trajanjem | Taskovi + dependencies + milestones |
| Resources | Nema resource plan | Djelomičan resource plan | Kompletan resource plan bez overallocation |

### **Razumijevanje Budgetiranja:**
| Kriterij | Početnik | Srednji | Napredni |
|----------|----------|---------|----------|
| Metode | Poznaje 1 metodu | Razlikuje 2-3 metode | Zna kada koristiti koju metodu |
| Three-Point | Ne može izračunati | Izračunava uz pomoć | Samostalno izračunava |
| Contingency | Ne uključuje | Uključuje ali ne objašnjava | Uključuje i objašnjava zašto |

## 🔄 **BACKUP AKTIVNOSTI**

### **BACKUP AKTIVNOST 1: WBS Card Sorting (15 min)**

**Cilj:** Studentice vježbaju grupiranje taskova u faze koristeći Post-it notes

**Kada koristiti:**
- Ako WBS vježba završi ranije
- Ako neka grupa ima problema s hijerarhijom

**Upute:**
1. Svaka osoba dobije 10 Post-it notes
2. Napišite 10 taskova za imaginarni projekt (npr. "Wedding Planning")
3. Sortirajte u logične faze
4. Predstavite svoj WBS

📚 **FACILITATOR NOTES:**
- Ovo je zabavna, hands-on verzija WBS kreiranja
- Koristite poznat projekt (wedding, house renovation) da bude relatable
- Nema "krivog" odgovora - različite osobe mogu imati različite strukture

---

### **BACKUP AKTIVNOST 2: Schedule Timeline Drawing (10 min)**

**Cilj:** Vizualizacija schedule-a bez formalnog Gantt Charta

**Kada koristiti:**
- Ako je Gantt Chart template zbunjujuć
- Za vizualne učenike

**Upute:**
1. Nacrtajte vremensku liniju (1-8 mjeseci)
2. Stavite svoje faze kao "blokove" na liniju
3. Označite milestones s ◆
4. Povežite strelicama dependencies

📚 **FACILITATOR NOTES:**
- Jednostavnija alternativa formalnom Gantt Chartu
- Fokus na razumijevanje KONCEPTA, ne na alat
- Može se raditi na whiteboard kao grupna aktivnost

---

### **BACKUP AKTIVNOST 3: Budget Debate (10 min)**

**Cilj:** Diskusija o prednostima i manama različitih metoda

**Kada koristiti:**
- Nakon budget vježbe
- Ako ima vremena za diskusiju

**Upute:**
1. Podijelite grupu u 3 tima
2. Svaki tim "brani" svoju metodu (Top-Down, Bottom-Up, Three-Point)
3. 2 min za pripremu argumenata
4. 1 min prezentacija po timu
5. Glasanje: Koja metoda je najbolja za EduTrack?

📚 **FACILITATOR NOTES:**
- Potiče kritičko razmišljanje
- Nema "ispravnog" odgovora - sve metode imaju svoje mjesto
- Zaključak: U praksi se često kombiniraju metode

---

### **BACKUP AKTIVNOST 4: WBS Critique (10 min)**

**Cilj:** Analizirati "loš" WBS i predložiti poboljšanja

**Kada koristiti:**
- Za naprednije grupe
- Ako ostane vremena nakon WBS vježbe

**Primjer lošeg WBS-a:**
```
EduTrack
├── 1.0 Stuff to do
├── 2.0 More stuff
├── 3.0 Technical things
└── 4.0 Finish
```

**Upute:**
1. Prikažite loš WBS
2. Pitajte: "Što je krivo s ovim WBS-om?"
3. Grupno identificirajte probleme
4. Predložite poboljšanja

📚 **FACILITATOR NOTES:**
Problemi: Nejasni nazivi, nema numeracije Level 3, nedostaje detalj

---

### **BACKUP AKTIVNOST 5: Resource Conflict Scenario (15 min)**

**Cilj:** Riješiti situaciju kada su resursi preopterećeni

**Kada koristiti:**
- Ako Gantt Chart vježba završi ranije
- Za diskusiju o resource management-u

**Scenarij:**
"Developer Ana je assigned na 3 taska koji se preklapaju:
- Frontend (tjedan 5-8)
- Backend API (tjedan 6-9)
- Database setup (tjedan 7-8)

Kako riješiti ovaj problem?"

**Upute:**
1. Prikažite scenarij
2. Grupe predlažu rješenja (5 min)
3. Prezentacija rješenja (5 min)
4. Diskusija o trade-offs (5 min)

📚 **FACILITATOR NOTES:**
Moguća rješenja:
- Sekvencijalizirati taskove (produljuje timeline)
- Dodati drugog developera (povećava budget)
- Paralelizirati gdje je moguće
- Prioritizirati critical path taskove

### **BACKUP AKTIVNOST 6: M3 Quick Quiz (10 min)**

**Cilj:** Provjera razumijevanja cijelog M3 modula

**Kada koristiti:**

- Za recap prije zaključka

**Pitanja:**
1. Koliko komponenti ima PMP? (10)
2. Koji su 5 elemenata Scope Statement-a? (Objective, Deliverables, Boundaries, Constraints, Acceptance Criteria)
3. Što je 100% Rule u WBS-u? (WBS mora pokrivati 100% scope-a)
4. Što je razlika između Lead i Lag? (Lead = ranije, Lag = čekanje)
5. Koja je formula za Three-Point? ((O+4M+P)/6)

📚 **FACILITATOR NOTES:**
- Može biti timsko natjecanje
- Bodovanje: 2 boda po pitanju
- Nagrada: Čokolada ili pohvala! 🏆

### **BACKUP AKTIVNOST 7: EduTrack Integration Exercise (20 min)**

**Cilj:** Integrirati WBS, Schedule i Budget u cjeloviti mini-plan

**Kada koristiti:**

📚 **FACILITATOR NOTES:**
- Ovo je "mini-PM plan"

- Odličan zaključak za M3 modul
