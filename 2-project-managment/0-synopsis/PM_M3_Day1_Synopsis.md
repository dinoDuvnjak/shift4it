# ⏰ **9:15-9:45 (30 min) - ŠTO JE PROJECT MANAGEMENT PLAN?**

### **Slajd 4: Project Management Plan**

**Notes iz PDF-a:**
_"A Project Management Plan is a formal document that defines how a project will be executed, monitored, and controlled. It consists of several key components."_

## **Što reći studenticama (10 min):**

"Sada ulazimo u srž planiranja - **Project Management Plan (PMP)**.

📋 **ŠTO JE PROJECT MANAGEMENT PLAN?**

**Definicija:**
> Project Management Plan je formalni dokument koji definira KAKO će se projekt izvršiti, pratiti i kontrolirati.

**Jednostavnije rečeno:**
PMP je "master dokument" projekta - svi ostali planovi i dokumenti su dio njega.

**Analogija:**
Ako je projekt izgradnja kuće:
- **Project Charter** = Dozvola za gradnju (autorizacija da možeš graditi)
- **Project Management Plan** = Nacrt kuće + upute za gradnju + raspored radova + budget

*[Nacrtaj na whiteboard]*
```
┌─────────────────────────────────────────────┐
│         PROJECT MANAGEMENT PLAN             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │  Scope  │  │Schedule │  │  Cost   │     │
│  │  Plan   │  │  Plan   │  │  Plan   │     │
│  └─────────┘  └─────────┘  └─────────┘     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ Quality │  │Resource │  │Communic.│     │
│  │  Plan   │  │  Plan   │  │  Plan   │     │
│  └─────────┘  └─────────┘  └─────────┘     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │  Risk   │  │Procurem.│  │Stakehol.│     │
│  │  Plan   │  │  Plan   │  │  Plan   │     │
│  └─────────┘  └─────────┘  └─────────┘     │
└─────────────────────────────────────────────┘
```

❓ **Razlika Project Charter vs. PMP:**

| Aspekt | Project Charter | Project Management Plan |
|--------|-----------------|------------------------|
| **Kada?** | Na početku (Initiation) | Tijekom planiranja |
| **Svrha** | Autorizira projekt | Definira kako izvršiti |
| **Detalji** | High-level | Detaljni |
| **Veličina** | 1-3 stranice | 10-100+ stranica |
| **Tko piše?** | Sponsor + PM | PM + Tim |

💡 **Ključna poruka:**
Charter pokreće projekt, PMP ga vodi do cilja!"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
PMP nije jedan dokument već SKUP dokumenata (sub-plans). U malim projektima može biti 5-10 stranica, u velikim enterprise projektima može imati stotine stranica.

**Česta zabuna:**
Studentice često misle da je PMP = raspored projekta. NE! Raspored (Schedule) je samo JEDAN dio PMP-a.

**Česta pitanja:**

Q: "Mora li svaki projekt imati PMP?"
A: "Da, ali razina detalja ovisi o veličini projekta. Mali projekt = jednostavan PMP. Veliki projekt = opsežan PMP."

Q: "Tko odobrava PMP?"
A: "Project Sponsor odobrava, ali PM piše uz pomoć tima i stakeholdera."

### **Slajd 5-6: Components of a Project Management Plan I, II, III**

**Notes iz PDF-a:**
_"PMP sadrži: 1) Project Overview, 2) Scope Management Plan, 3) Schedule Management Plan, 4) Cost Management Plan, 5) Quality Management Plan, 6) Resource Management Plan, 7) Communication Management Plan, 8) Risk Management Plan, 9) Procurement Management Plan, 10) Stakeholder Management Plan."_

## **Što reći studenticama (20 min):**

"PMP ima **10 ključnih komponenti**. Idemo redom:

---

📊 **1. PROJECT OVERVIEW**

**Što uključuje:**
- Project Name & Description - naziv i opis projekta
- Objectives & Goals - ciljevi projekta
- Project Scope - visoki pregled scope-a
- Success Criteria - kako mjerimo uspjeh

*IT Primjer:*
'HealthTrack Mobile App - aplikacija za praćenje fitnessa. Cilj: lansiranje za 6 mjeseci s 10.000 download-a.'

---

📝 **2. SCOPE MANAGEMENT PLAN**

**Što uključuje:**
- Kako će se scope definirati i kontrolirati
- Procesi za scope changes
- Sprječavanje scope creep-a (neovlaštenih promjena)

*IT Primjer:*
'Svaka promjena scope-a zahtijeva Change Request Form i odobrenje Sponsora.'

💡 **Scope creep** = kad projekt polako raste bez kontrole. "Možemo li dodati još jednu malu funkcionalnost?" × 50 = projekt nikad ne završi!

---

📅 **3. SCHEDULE MANAGEMENT PLAN**

**Što uključuje:**
- Project schedule (Gantt chart, milestones)
- Task dependencies - koji task ovisi o kojem
- Metode praćenja napretka

*IT Primjer:*
'Development faza: Mjesec 2-4. Milestone: Alpha verzija do 15. ožujka.'

---

💰 **4. COST MANAGEMENT PLAN**

**Što uključuje:**
- Initial budget estimate
- Cost tracking - praćenje troškova
- Variance analysis - analiza odstupanja

*IT Primjer:*
'Ukupni budget: 150.000€. Development: 80.000€. Design: 30.000€. Testing: 20.000€. Marketing: 20.000€.'

---

✅ **5. QUALITY MANAGEMENT PLAN**

**Što uključuje:**
- Quality objectives - ciljevi kvalitete
- Testing procedures - procedure testiranja
- Quality control measures - mjere kontrole kvalitete

*IT Primjer:*
'Max 5 critical bugs-a na lansiranju. Code review obavezan prije merge-a.'

---

👥 **6. RESOURCE MANAGEMENT PLAN**

**Što uključuje:**
- Roles & responsibilities - tko radi što
- Hiring and training needs - potrebe za zapošljavanjem
- Resource allocation - raspodjela resursa

*IT Primjer:*
'Tim: 3 developera, 1 designer, 1 QA. Senior developer potreban od mjeseca 2.'

---

📢 **7. COMMUNICATION MANAGEMENT PLAN**

**Što uključuje:**
- Communication methods - email, meeting, Slack
- Frequency of updates - koliko često
- Stakeholder communication matrix

*IT Primjer:*
'Weekly standup: ponedjeljak 9:00. Monthly report Sponsoru. Slack za dnevnu komunikaciju.'

---

⚠️ **8. RISK MANAGEMENT PLAN**

**Što uključuje:**
- Risk identification - prepoznavanje rizika
- Risk response strategies - strategije odgovora (avoid, mitigate, transfer, accept)
- Contingency plans - planovi za nepredviđene situacije

*IT Primjer:*
'Rizik: Key developer napusti projekt. Mitigacija: Dokumentacija, cross-training.'

---

🛒 **9. PROCUREMENT MANAGEMENT PLAN**

**Što uključuje:**
- Vendor selection criteria - kriteriji za odabir dobavljača
- Contract types - tipovi ugovora
- Make-or-buy decisions - napraviti sami ili kupiti

*IT Primjer:*
'AWS za hosting (3-year reserved instance). Figma licence za design tim.'

---

👤 **10. STAKEHOLDER MANAGEMENT PLAN**

**Što uključuje:**
- Stakeholder identification - već smo radili u M2!
- Engagement strategies - strategije angažiranja
- Communication preferences

*IT Primjer:*
'CEO: Monthly executive summary. Dev team: Daily standup. Users: Beta feedback sessions.'

---

💡 **ZAKLJUČAK:**

Svih 10 komponenti je POVEZANO! Promjena u jednom utječe na ostale.

*[Nacrtaj trokut na whiteboard]*
```
        SCOPE
         /\
        /  \
       /    \
      /______\
   TIME     COST
```

**Triple Constraint u PMP-u:**
- Povećaš SCOPE → trebaš više TIME ili COST
- Smanjiš TIME → moraš smanjiti SCOPE ili povećati COST
- Smanjiš COST → moraš smanjiti SCOPE ili produljiti TIME

Ovo je srž project managementa!"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
10 komponenti može zvučati overwhelming, ali u praksi se međusobno preklapaju. Quality plan utječe na Cost plan (više testiranja = više troškova). Risk plan utječe na Schedule (contingency vrijeme).

**Kako zapamtiti 10 komponenti:**
Akronim: **S S C Q R C R P S** + Overview
- **S**cope, **S**chedule, **C**ost, **Q**uality, **R**esource, **C**ommunication, **R**isk, **P**rocurement, **S**takeholder

**Česta pitanja:**

Q: "Mora li PMP imati svih 10 komponenti?"
A: "Za male projekte - ne sve. Ali scope, schedule i cost su UVIJEK potrebni."

Q: "Koliko detaljno treba biti?"
A: "Dovoljno da netko tko nije bio uključen može razumjeti projekt. Ni previše, ni premalo."

Q: "Tko piše koji dio?"
A: "PM koordinira, ali dijelovi se pišu s odgovornim osobama. Tech Lead piše tehnički dio, QA piše Quality plan, itd."

📖 **Reference:**
- [PMBOK Guide - PMP Components](https://www.pmi.org/pmbok-guide-standards)

# ⏰ **9:45-10:10 (25 min) - PMP EXAMPLE: HEALTHTRACK MOBILE APP**

### **Slajdovi 7-11: Project Management Plan Example**

**Notes iz PDF-a:**
_"HealthTrack Mobile App - Develop a mobile app for tracking fitness and health metrics. Features: step tracking, calorie counting, workout logging."_

## **Što reći studenticama (25 min):**

"Teorija je super, ali najbolje se uči na primjeru! Idemo kroz realan PMP za **HealthTrack Mobile App**.

📱 **CASE STUDY: HealthTrack Mobile App**

*[Pročitaj case study s osjećajem kao da prezentiraš pravi projekt]*

---

**1. PROJECT OVERVIEW**

| Element | Detalji |
|---------|---------|
| **Project Name** | HealthTrack Mobile App |
| **Objective** | Razviti mobilnu aplikaciju za praćenje fitnessa i zdravstvenih metrika |
| **Scope** | Design, development i launch Android + iOS aplikacije |
| **Features** | Step tracking, calorie counting, workout logging |
| **Success Criteria** | Launch on time, meets user requirements, 10.000 downloads u prva 3 mjeseca |

💡 **Za diskusiju:**
'Što mislite - je li success criteria od 10.000 download-a realna? Kako biste vi definirali uspjeh za fitness app?'

---

**2. SCOPE MANAGEMENT PLAN**

- Scope uključuje: UI/UX design, app development, backend integration, testing
- Scope NE uključuje: Marketing (odvojen projekt), Support nakon lansiranja (operations)
- **Change Control:** Nove feature zahtjeve ide kroz Change Request proces

💡 **Scope boundaries su ključne!**
Ovdje jasno piše da marketing NIJE dio ovog projekta. Bez toga bi netko mogao reći 'zašto nismo napravili TikTok kampanju?' - odgovor: 'Nije bilo u scope-u.'

---

**3. SCHEDULE MANAGEMENT PLAN**

**Milestones:**
| Milestone | Datum |
|-----------|-------|
| Design completion | Mjesec 1 |
| Development | Mjesec 2-4 |
| Testing | Mjesec 5 |
| Launch | Mjesec 6 |

**Tracking:** Weekly progress updates via JIRA

💡 **Za diskusiju:**
'Ako development kasni 2 tjedna, što se događa s testingom i launchom? Kako biste reagirali?'

---

**4. COST MANAGEMENT PLAN**

**Estimated Budget: 150.000€**

| Kategorija | Iznos |
|------------|-------|
| Development | 80.000€ |
| Design | 30.000€ |
| Testing | 20.000€ |
| Marketing | 20.000€ |

**Control Measures:** Monthly budget reviews i variance analysis

💡 **Primjetite:**
- Development je 53% budgeta - normalno za software projekt
- Testing je 'samo' 13% - je li to dovoljno? Ovisi o kompleksnosti!

---

**5. RISK MANAGEMENT PLAN**

| Rizik | Mitigation Strategy |
|-------|---------------------|
| Delay in development | Use Agile methodology for flexibility |
| Budget overrun | Regular financial monitoring |
| Low user adoption | Conduct beta testing and gather feedback |

💡 **Za diskusiju:**
'Koji još rizici postoje za mobilnu aplikaciju? Što ako Apple odbije app na App Store?'

---

**6. COMMUNICATION MANAGEMENT PLAN**

| Tip komunikacije | Frekvencija | Alat |
|------------------|-------------|------|
| Team meetings | Weekly stand-ups | Zoom |
| Stakeholder updates | Monthly reports | Email |
| Daily communication | Continuous | Slack |
| Task tracking | Continuous | JIRA |

---

**7. STAKEHOLDER MANAGEMENT PLAN**

| Uloga | Odgovornost |
|-------|-------------|
| Project Sponsor | Funding and oversight |
| Project Manager | Execution and risk management |
| Developers | Coding and technical implementation |
| QA Team | Quality testing before launch |

---

🎯 **ANALIZA - Što možemo naučiti iz ovog PMP-a?**

**Pozitivne strane:**
✅ Jasno definirani milestones
✅ Specifični success criteria (10.000 downloads)
✅ Risk mitigation strategije
✅ Jasna komunikacijska struktura

**Što bi se moglo poboljšati:**
⚠️ Nema detaljnog resource plan-a (koliko developera?)
⚠️ Quality plan nije detaljno opisan
⚠️ Nema Procurement plan-a (koji tools/services?)

**Pitanja za razmišljanje:**
- Što ako je 80.000€ premalo za development?
- Tko je backup ako key developer ode?
- Kako mjerimo 'user satisfaction'?"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
Ovaj PMP primjer je pojednostavljen za edukaciju. U stvarnosti bi bio detaljniji (npr. Schedule bi imao Gantt chart s taskovima, ne samo milestones).

### **Slajd 12: Group Exercise - Change Management**

**Notes iz PDF-a:**
_"Based on the PMP described above the trainees will: 1) Recognize the impact of changes in one area on the overall project. 2) Develop a change management plan to handle scope changes and their impact on schedule and budget."_

## ✏️ **AKTIVNOST: Kreiranje Change Management Plan-a**

**Cilj:** Studentice razumiju kako promjene u projektu utječu na scope, schedule i budget, te kreiraju proces za upravljanje promjenama

**Trajanje:** 35 min

**Materijali:** 
- Change Request Form Template (handout)
- Flip chart papir za svaku grupu
- Markeri (4 boje)

### **Kontekst (3 min):**

*Reci studenticama:*

"Zamislite sljedeći scenarij:

📱 **HealthTrack projekt, Mjesec 3 od 6**

Projekt je u tijeku. Development je završen 60%. Stigao je zahtjev od marketinga:

> *'Trebamo integrirati aplikaciju s Apple Watch i Fitbit uređajima. Konkurencija to ima, moramo i mi!'*

Vaš zadatak kao PM tima:
1. Procijeniti utjecaj ove promjene
2. Kreirati Change Request Process
3. Odlučiti - prihvaćamo li promjenu?

---

# ⏰ **11:15-11:35 (20 min) - SCOPE STATEMENT - TEORIJA + PRIMJER**

### **Slajd 14: How to Write a Clear and Concise Scope Statement**

**Notes iz PDF-a:**
_"A well-defined Scope Statement should include: 1) Project Objective, 2) Deliverables, 3) Scope Boundaries, 4) Constraints & Assumptions, 5) Acceptance Criteria."_

## **Što reći studenticama (12 min):**

"Sada ulazimo u jedan od NAJVAŽNIJIH dokumenata u planiranju - **Scope Statement**.

📝 **ŠTO JE SCOPE STATEMENT?**

**Definicija:**
> Scope Statement je dokument koji jasno definira ŠTO je uključeno u projekt, a ŠTO NIJE.

**Zašto je toliko važan?**
- Sprječava **scope creep** (nekontrolirano širenje projekta)
- Svi stakeholderi imaju ISTO razumijevanje projekta
- Osnova za planiranje schedule-a i budgeta

**Analogija:**
Scope Statement je poput menija u restoranu:
- Jasno piše što možete naručiti (uključeno)
- Jasno piše što NE nudimo (isključeno)
- Piše cijena i vrijeme pripreme (constraints)

---

🎯 **5 ELEMENATA DOBROG SCOPE STATEMENT-a:**

*[Nacrtaj na whiteboard]*

```
┌─────────────────────────────────────────┐
│         SCOPE STATEMENT                 │
├─────────────────────────────────────────┤
│ 1. PROJECT OBJECTIVE                    │
│    Što projekt želi postići?            │
├─────────────────────────────────────────┤
│ 2. DELIVERABLES                         │
│    Koji su konkretni outputi?           │
├─────────────────────────────────────────┤
│ 3. SCOPE BOUNDARIES                     │
│    Što JE uključeno? Što NIJE?          │
├─────────────────────────────────────────┤
│ 4. CONSTRAINTS & ASSUMPTIONS            │
│    Ograničenja vremena, budgeta, resursa│
├─────────────────────────────────────────┤
│ 5. ACCEPTANCE CRITERIA                  │
│    Kako znamo da smo uspjeli?           │
└─────────────────────────────────────────┘
```

---

**1. 🎯 PROJECT OBJECTIVE**
Jasno navedi ŠTO projekt želi postići.

✅ **Dobro:** 'Razviti mobilnu aplikaciju za praćenje fitnessa koja omogućuje korisnicima praćenje koraka, kalorija i treninga.'

❌ **Loše:** 'Napraviti super app.'

---

**2. 📦 DELIVERABLES**
Nabroji konkretne outpute projekta.

✅ **Dobro:**
- iOS aplikacija
- Android aplikacija
- Backend API
- Admin dashboard
- Korisnička dokumentacija

❌ **Loše:** 'Aplikacija i sve što treba.'

---

**3. 🚧 SCOPE BOUNDARIES**
Eksplicitno navedi što JE i što NIJE uključeno.

✅ **Dobro:**
- **UKLJUČENO:** Step tracking, calorie counting, workout logging
- **ISKLJUČENO:** Diet planning, social features, wearable integrations

❌ **Loše:** Samo navesti što je uključeno bez isključenja.

💡 **Zašto pisati što NIJE uključeno?**
Jer stakeholderi često PRETPOSTAVLJAJU da je nešto uključeno. Ako jasno napišeš 'diet planning NIJE uključen', nitko ne može kasnije reći 'mislio sam da to radimo'.

---

**4. ⚠️ CONSTRAINTS & ASSUMPTIONS**
Navedi ograničenja i pretpostavke.

**Constraints (Ograničenja):**
- Time: Projekt mora biti gotov za 6 mjeseci
- Budget: Maksimalno 150.000€
- Resources: Dostupan tim od 5 osoba

**Assumptions (Pretpostavke):**
- Apple i Google neće promijeniti API
- Korisnici imaju iOS 14+ ili Android 10+
- AWS će biti dostupan 99.9% vremena

---

**5. ✅ ACCEPTANCE CRITERIA**
Definiraj kako se mjeri uspjeh.

✅ **Dobro:**
- App funkcionira bez critical bugs-a
- User satisfaction score > 4.0/5.0
- 10.000 downloads u prva 3 mjeseca
- Response time < 2 sekunde

❌ **Loše:** 'Korisnici su zadovoljni.'"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
Scope Statement je "ugovor" između PM-a i stakeholdera. Kad netko traži nešto što nije u scope-u, možeš reći: "To nije bilo u Scope Statement-u. Možemo ga dodati kroz Change Request proces."

**Česta greška:**
Scope Statement koji je previše općenit. "Napraviti dobar softver" nije scope - to je želja. Scope mora biti SPECIFIČAN i MJERLJIV.

**Česta pitanja:**

Q: "Što ako se Scope promijeni tijekom projekta?"
A: "To je normalno! Koristi Change Request Process koji smo upravo učili. Ažuriraj Scope Statement kad se promjena odobri."

Q: "Tko piše Scope Statement?"
A: "PM, uz input od tehničkog tima i stakeholdera. Sponsor odobrava."

Q: "Koliko detaljan treba biti?"
A: "Dovoljno da svatko tko čita razumije granice projekta. Ako se možeš pitati 'je li ovo uključeno?', odgovor treba biti u dokumentu."

### **Slajd 15: Scope Statement Example**

**Notes iz PDF-a:**
_"HealthTrack primjer Scope Statement-a."_

## **Što reći studenticama (8 min):**

"Evo primjera dobrog Scope Statement-a:

---

📱 **SCOPE STATEMENT: HealthTrack Mobile App**

> *'The HealthTrack Mobile App project will develop a fitness tracking app for iOS and Android, featuring step counting, calorie tracking, and workout logging. The project will NOT include diet planning features. The final product must be delivered within 6 months, with a budget of $150,000. Success will be measured by app functionality, user satisfaction, and at least 10,000 downloads in the first three months.'*

---

🔍 **ANALIZA - Pronađimo 5 elemenata:**

| Element | Tekst iz primjera |
|---------|-------------------|
| **1. Objective** | 'develop a fitness tracking app for iOS and Android' |
| **2. Deliverables** | 'step counting, calorie tracking, workout logging' |
| **3. Boundaries** | 'will NOT include diet planning features' |
| **4. Constraints** | '6 months, $150,000' |
| **5. Acceptance Criteria** | 'app functionality, user satisfaction, 10,000 downloads' |

✅ **Što je dobro:**
- Jasno navodi ŠTO se radi
- Eksplicitno kaže što NIJE uključeno
- Specifični brojevi (6 mjeseci, 150k$, 10k downloads)

⚠️ **Što bi se moglo poboljšati:**
- Dodati više detalja o deliverables-ima (iOS + Android verzija, backend?)
- Definirati 'user satisfaction' preciznije (npr. rating > 4.0)
- Navesti više boundaries (npr. 'no social features', 'no wearable integrations')

💡 **Zaključak:**
Čak i 'dobar' Scope Statement može biti bolji. Cilj je biti što precizniji da spriječimo nesporazume!"

📚 **FACILITATOR NOTES:**

**Za tvoje razumijevanje:**
Ovaj primjer je kratak (1 paragraf) što je OK za manje projekte. Za velike projekte Scope Statement može biti 2-5 stranica s detaljnim listama deliverables-a i isključenja.

**Česta pitanja:**

Q: "Je li Scope Statement isto što i Project Charter scope sekcija?"
A: "Slično, ali Scope Statement je DETALJNIJI. Charter ima high-level scope, Scope Statement ima puni detalj."

## 🔄 **BACKUP AKTIVNOSTI**

### **BACKUP AKTIVNOST 1: PMP Component Matching Game (10 min)**

**Cilj:** Brza provjera razumijevanja 10 PMP komponenti

**Kada koristiti:**
- Ako glavna aktivnost završi prerano
- Kao energizer nakon pauze
- Za učvršćivanje gradiva na kraju dana

**Materijali:** Kartice s komponentama i opisima (pripremi unaprijed)

**Upute:**
1. Napiši na whiteboard 10 komponenti PMP-a
2. Pročitaj opise jedan po jedan
3. Studentice pogađaju koja komponenta odgovara opisu

**Primjeri opisa:**
- "Ovaj plan definira kako ćemo osigurati da nema bug-ova" → Quality Management Plan
- "Ovaj plan navodi tko koristi Slack, a tko email" → Communication Management Plan
- "Ovaj plan identificira što može poći po zlu" → Risk Management Plan

📚 **FACILITATOR NOTES:**
Napravi igru natjecateljskom - grupe natječu tko prvi pogodi. Dodaj bodove za dodatnu motivaciju.

---

### **BACKUP AKTIVNOST 2: Scope Creep Scenario Discussion (15 min)**

**Cilj:** Razumijevanje opasnosti nekontroliranog širenja scope-a

**Kada koristiti:**
- Nakon Scope Statement teorije
- Ako ostane vremena prije pauze

**Upute:**
1. Predstavi scenarij:
   > *"Radite na EduTrack projektu. Ovo su zahtjevi koje ste dobili tijekom 3 mjeseca:"*
   > - Tjedan 2: "Možemo li dodati dark mode?"
   > - Tjedan 4: "Trebamo integraciju s Google Classroom"
   > - Tjedan 6: "Roditelji žele mobile app"
   > - Tjedan 8: "Možemo li dodati video calling?"
   > - Tjedan 10: "Trebamo AI tutor funkciju"

2. Pitaj studentice:
   - Koje zahtjeve biste prihvatili?
   - Koje biste odbili?
   - Kako biste komunicirali odbijanje?

📚 **FACILITATOR NOTES:**
Cilj je pokazati da bez Scope Statement-a, projekt nikad ne završava. "Svaki 'mali' zahtjev dodaje vrijeme i novac!"

---

### **BACKUP AKTIVNOST 3: Triple Constraint Simulation (10 min)**

**Cilj:** Praktično razumijevanje međuovisnosti scope, time i cost

**Kada koristiti:**
- Na početku za warm-up
- Kao vizualizacija koncepta

**Materijali:** Trokut nacrtan na whiteboard

**Upute:**
1. Nacrtaj trokut s Scope, Time, Cost
2. Daj scenarije i pitaj što se mijenja:
   - "Klijent želi 3 dodatne funkcionalnosti" → Scope ↑, što se događa?
   - "Budget je smanjen za 20%" → Cost ↓, što se događa?
   - "Launch mora biti 2 tjedna ranije" → Time ↓, što se događa?

📚 **FACILITATOR NOTES:**
Koristite fizičke geste - "ako povučem ovaj kut, ostala dva se pomiču". Vizualizacija pomaže razumijevanju.

---

### **BACKUP AKTIVNOST 4: Change Request Role Play (15 min)**

**Cilj:** Vježbanje komunikacije oko change request-a

**Kada koristiti:**
- Umjesto ili kao dodatak Change Management aktivnosti
- Za grupe koje vole role play

**Upute:**
1. Podijeli uloge:
   - PM (brani scope)
   - Stakeholder (traži promjenu)
   - Sponsor (donosi konačnu odluku)

2. Scenarij: Stakeholder traži veliku promjenu usred projekta

3. Svaka uloga ima 2 minute da prezentira svoj stav

4. Sponsor donosi odluku i objašnjava zašto

📚 **FACILITATOR NOTES:**
Pripremi kartice s ulogama i "script points" za svaku ulogu. Naglasi da nema "pogrešnog" odgovora.

---

### **BACKUP AKTIVNOST 5: Scope Statement Critique (10 min)**

**Cilj:** Prepoznavanje slabosti u Scope Statement-u

**Kada koristiti:**
- Nakon Scope Statement vježbe
- Za napredniju diskusiju

**Materijali:** Loš primjer Scope Statement-a

**Upute:**
1. Prikaži "loš" Scope Statement:
   > *"We will build a great app that users will love. It will have cool features and modern design. We aim to finish it soon and make everyone happy."*

2. Pitaj studentice što nedostaje

3. Zajedno popravite

**Očekivani problemi:**
- Nema specifičnih deliverables-a
- Nema boundaries (što nije uključeno)
- Nema constraints (vrijeme, budget)
- Acceptance criteria nisu mjerljivi

📚 **FACILITATOR NOTES:**
Ovaj "loš" primjer je namjerno exaggerated. U praksi su loši Scope Statements manje očiti, ali imaju iste probleme.

---

### **BACKUP AKTIVNOST 6: M1-M2-M3 Connection Map (10 min)**

**Cilj:** Povezivanje znanja kroz module

**Kada koristiti:**
- Na kraju dana za recap
- Ako studentice imaju poteškoća s povezivanjem koncepata

**Upute:**
1. Nacrtaj na whiteboard:
   ```
   M1 (Fundamentals) → M2 (Starting) → M3 (Planning)
   ```

2. Pitaj studentice da povežu koncepte:
   - Triple Constraint (M1) → PMP (M3)
   - Project Charter (M2) → Scope Statement (M3)
   - Stakeholder Roles (M2) → Stakeholder Management Plan (M3)

📚 **FACILITATOR NOTES:**
Ova aktivnost pomaže studentima vidjeti "big picture" i razumjeti zašto smo radili što smo radili.

---

### **BACKUP AKTIVNOST 7: Quick Quiz - True or False (5 min)**

**Cilj:** Brza provjera razumijevanja

**Kada koristiti:**
- Na kraju dana
- Između sekcija za energizer

**Pitanja:**

1. "PMP ima 5 komponenti." → FALSE (ima 10)
2. "Scope Statement definira što NIJE uključeno u projekt." → TRUE
3. "Change Request uvijek mora odobriti Sponsor." → FALSE (minor changes može PM)
4. "Scope creep je uvijek loš." → TRUE (nekontrolirano širenje je problem)
5. "PMP se piše jednom i nikad ne mijenja." → FALSE (living document)

📚 **FACILITATOR NOTES:**
Neka studentice dignnu ruku za TRUE, spuste za FALSE. Brzo i interaktivno!

## ✅ **PITANJA ZA PROVJERU RAZUMIJEVANJA**

### **Osnovni level:**

1. Što je Project Management Plan?
2. Koliko komponenti ima PMP?
3. Koja 3 elementa čine Triple Constraint?
4. Kojih 5 elemenata ima dobar Scope Statement?

### **Srednji level:**

5. Koja je razlika između Project Charter-a i PMP-a?
6. Zašto je važno definirati što NIJE uključeno u scope?
7. Opišite proces Change Request-a.
8. Što se događa ako povećamo scope bez povećanja budgeta ili vremena?

### **Napredni level:**

9. Kako biste odlučili da li prihvatiti Change Request koji povećava scope za 30%?
10. Navedite primjer kako Quality Management Plan utječe na Cost Management Plan.
11. Kako biste napisali Scope Statement za projekt koji još nije jasno definiran?
